# Chaos Mesh Experiment: CPU Stress Testing with Concert Resilience Scoring

## Overview

This document describes a chaos engineering experiment conducted using Chaos Mesh to test system resilience under CPU stress conditions. The experiment leverages Concert's auto-discovery feature to measure and track resilience scores before and after introducing chaos and get relevant actions to remediate it.

## Experiment Objectives

- Perform Concert Auto-Discovery before chaos injection to establish a baseline resilience score
- Introduce controlled chaos through stress tests
- Re-run Auto-Discovery after chaos injection to observe degradation in resilience
- Analyze the resilience score changes and actions generated by Concert
- Demonstrate how Concert’s resilience features help identify weaknesses under stress

## Prerequisites

- Kubernetes cluster (v1.19+)
- Chaos Mesh installed and configured
- Concert(>v2.2.0) with watsonX enabled
- chaos_test Profile which includes the 'workloads within 90% of cpu limit' metric
- Appropriate RBAC permissions for chaos experiments

## Setup: Creating the chaos_test Profile

Before running the experiment, you need to create a custom profile that includes the 'workloads within 90% of cpu limit' metric. Follow these steps:

1. **Login to Concert UI**
   - Access the Concert web interface with your credentials

2. **Navigate to Resilience Dimension**
   - Go to the **Resilience** dimension from the main navigation

3. **Access Profiles Tab**
   - Click on the **Profiles** tab

4. **Create New Profile**
   - Click on **Create Profile** button

5. **Enter Profile Name and description**
   - Set the profile name as: `chaos_test`
   - Provide a meaningful description (e.g., "Profile for chaos engineering experiments with CPU stress testing")
   - Click **Next**

6. **Select Library, Catefory and Requirement**
   - Click on the **Library** radio button
   - Select **Runtime Production Readiness Library** from the dropdown
   - In the **Categories** section, select **Integrity**
   - In the **Requirements** section, select **"Workloads Within 90% of CPU Limit"**
   - Click **Next**

7. **Create Profile**
    - Review the configuration
    - Click **Create** to finalize the profile


## Experiment Workflow

This section outlines the exact reference flow followed to demonstrate Concert resilience behavior for a more than 90% CPU stress scenario.


### Step 1: Chaos_test Profile(90_cpu mteric)

- Identify the chaos_test profile 
- Look for workloads within 90% of cpu limit metric in the profile
- This metric serves as the reference point for resilience scoring and actions


### Step 2: Run Initial Auto-Discovery (Before Chaos)

- Execute Concert Auto-Discovery on the target namespace
- Capture baseline resilience score


### Step 3: Verify Resilience Score for 90_cpu (Pre-Chaos)

- Navigate to Dimension → Resilience
- Locate the score associated with 90_cpu


### Step 4: Verify Actions (Pre-Chaos)

- Navigate to Actions
- Confirm that No 90% CPU–specific action is generated


### Step 5: Run Chaos Mesh Stress Test

- The chaos experiment can be executed using either of the following approaches.

1. Script-Based Execution
  - This approach is suitable for automation and repeatable testing.
  ```
  chmod +x run-chaos.sh
  ./run-chaos.sh
  ```
  - Experiment status can be verified using:
  ```
  oc get stresschaos -n chaos-mesh
  ```

2. Chaos Mesh Dashboard (UI)
  - Access the Chaos Mesh Dashboard using an OpenShift route (For example for the demo - http://chaos-dashboard-chaos-mesh.apps.fordocp.cp.fyre.ibm.com/)
  - Create a new Stress Fault experiment
  - Configure CPU, memory, duration, and target selectors as per cpu-memory-stress.yaml
  - Start the experiment and monitor execution
  - **Note:** If needed, you can pause the chaos experiment at any time from the dashboard by selecting the experiment and clicking the pause button. This allows you to temporarily halt the chaos injection without deleting the experiment configuration.



### Step 6: Run Auto-Discovery Again (During/After Chaos)

- While CPU stress is active (or immediately after), re-run Concert Auto-Discovery on the same namespace
- Concert reassesses the system under high CPU pressure


### Step 7: Verify Resilience Score for 90_cpu (Post-Chaos)

- Navigate to Resilience Score
- Locate the `workloads within 90% of cpu limit` metric again
- Observe the reduced resilience score compared to pre-chaos baseline


### Step 8: Verify Actions (Post-Chaos)

- Navigate to actions
- Confirm that 90% CPU–specific action is now generated


## Key Findings

### Resilience Score Changes

| Phase | Resilience Score | Status |
|-------|-----------------|--------|
| Pre-Chaos (Baseline) | Higher | ✅ Healthy |
| Post-Chaos (CPU Stress) | Lower | ⚠️ Degraded |

- **Before Chaos**
- Higher resilience score
- System operates within expected performance thresholds
- Fewer resilience risks identified

- **After Stress Test**
- Resilience score decreases
- Increased risk signals due to elevated CPU usage
- Degradation observed in performance-related NFRs


### Observed Impacts

- **CPU Stress Impact:**
  - Response time degradation
  - Increased latency
  - Potential service throttling
  - Resource contention issues


## Generated Actions

Concert generated the following recommendation based on the resilience score decrease:

Implement resource requests and limits for all workloads in Kubernetes to ensure efficient CPU utilization, and configure Horizontal Pod Autoscaling to scale workloads based on CPU usage, targeting a maximum utilization of 80% to avoid exceeding the 90% CPU limit threshold.

## Chaos Mesh Configuration

### Stress Experiment

```yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: StressChaos
metadata:
name: cpu-memory-stress
namespace: chaos-mesh
spec:
mode: all
selector:
namespaces:
- sample-app # change to your app namespace
labelSelectors:
app: sample-app # change to your app label
stressors:
cpu:
workers: 2
load: 98 # Load is 98%
memory:
workers: 2
size: "200" # Size is 200MB
duration: "30m"
```

## Key Observations

1. CPU stress leads to measurable resilience degradation
2. Concert effectively captures resilience changes via scoring
3. Auto-Discovery simplifies resilience assessment under chaos
4. Generated actions provide clear guidance for improving system robustness

## Safety Considerations

- Always run chaos experiments in non-production environments first
- Ensure proper monitoring and alerting is in place
- Have rollback procedures ready
- Communicate with team members before running experiments
- Start with low-impact experiments and gradually increase intensity
